# ADR 013 — 취소 가능 시간 기준: 확정 시각 → 공연 시작 시각 기준으로 변경 (초안)

**날짜**: 2026-07-09
**상태**: 결정 1 확정 · 결정 2(수수료 정책)는 착수 전
**관련 코드**: order-service `Order.isWithinCancellationWindow()`, `OrderProperties.Cancellation.cancellableWindowHours` — 코드 변경은 order-service 담당 파트 몫, 이 문서는 결정 사항 기록용

---

## 배경

order-service `Order.java:193-199`에 취소 가능 시간 판정 로직이 있는데, 현재는 **확정 시각(`statusUpdatedAt`) 기준 24시간**을 임시값으로 쓰고 있고 코드에 `// todo : 취소 가능 시간에 대해 협의 해야 함`이 그대로 남아 있다:

```java
public boolean isWithinCancellationWindow(long windowHours) {
    return this.statusUpdatedAt.plusHours(windowHours).isAfter(LocalDateTime.now());
}
```

이 방식은 "언제 예매했든 예매 후 24시간은 무조건 취소 가능"이라, 공연이 임박했는데도 취소가 열려있거나 반대로 공연이 한참 남았는데도 예매를 일찍 했다는 이유만으로 취소가 막히는 문제가 있다. ticketing-service `architecture.md §6`에도 "CONFIRMED 취소 시 좌석 처리 — 취소 가능 시간 기준이 order-service 쪽에서도 미정"으로 연동 미확정 항목으로 남아 있었다.

---

## 결정 1 (확정) — 기준을 "공연 시작 시각"으로 변경, 컷오프는 1시간 전

취소 가능 시간의 기준점을 `statusUpdatedAt`(확정 시각)이 아니라 **공연 시작 시각**으로 바꾼다. 공연 시작 1시간 전까지만 취소 가능하고, 그 이후는 취소 불가.

> **코드 반영 상태 (2026-07-09 확인)**: 이 리포에 체크아웃된 `Order.java:193-199`엔 아직 예전 로직(확정 시각 기준 + `todo: 협의 필요` 주석)이 남아있다. 결정 자체는 확정됐지만 코드 반영 여부/브랜치는 별도 확인 필요.

---

## 결정 2 (추후 확장 설계 예정, 착수 전) — 관람일 기준 단계별 취소 수수료

실제 운영 요구사항은 아래처럼 더 세분화된 수수료 정책이다. 이번 ADR에서 확정하는 건 아니고, 방향을 남겨두는 용도:

| 관람일 기준 | 수수료 |
|---|---|
| 9일 전까지 | 무료 |
| 8~7일 전 | 10% |
| 6~3일 전 | 20% |
| 2~1일 전 | 30% |
| 당일 | 취소 불가 |

- 취소 가능 마감은 원칙적으로 **공연일 1일 전까지**(공휴일/영업일 보정은 고려하지 않음)
- **예외**: 당일 예매 건은 예매한 그날(공연 당일) 안에서는 취소 가능해야 함 — "당일은 취소 불가" 규칙과 상충하므로, 판정 로직에 "예매 시각이 공연 당일인가"를 별도 분기로 둬야 함

---

## 이유

- **결정 1**: 확정 시각 기준은 "취소 여유가 예매 타이밍에 따라 들쭉날쭉해진다"는 근본 문제가 있고, 실제 취소 정책(수수료 포함)은 업계 관례상 전부 "공연일까지 며칠 남았는가"로 계산되므로 기준점을 공연 시작 시각으로 맞추는 게 이후 수수료 정책 확장과도 자연스럽게 이어진다.
- **결정 2**: 단일 컷오프(1시간)만으로는 실제 수수료 요구사항을 표현할 수 없다 — 구간별 수수료율을 나중에 얹을 걸 감안해서, 지금 시점엔 "1시간 컷오프"만 확정하고 수수료 계산 로직은 별도 설계로 분리한다(스코프 폭발 방지).

---

## 영향

- `isWithinCancellationWindow(long windowHours)`의 파라미터가 "확정 시각 기준 시간"에서 "공연 시작 기준 컷오프"로 의미가 바뀐다 — 시그니처 자체는 유지 가능하지만 호출부에서 넘기는 기준 시각이 `statusUpdatedAt`이 아니라 공연 시작 시각으로 바뀌어야 한다.
- `Order` 엔티티가 공연 시작 시각을 알아야 한다 — 현재 `Order`가 이 값을 갖고 있는지 확인 필요(없다면 조회 경로 추가 필요, order-service 담당 파트 확인 사항).
- 수수료 정책(결정 2)이 실제로 들어가면 `Payment`/환불 로직에 "부분 환불" 개념이 새로 필요해진다 — 지금은 전액 환불/거절 이진 구조라 스키마 변경 가능성 있음.
