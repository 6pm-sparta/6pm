# ADR 017 — inventory 최초 초기화에 Redisson 분산락 적용 (#362)

**날짜**: 2026-07-10 (수정 자체는 2026-07-08 — README에 흩어져 있던 기록을 소급 정리)
**상태**: 확정 (운영 중)

---

## 배경

`inventory:{showId}` 키는 쇼/좌석 생성 시점에 세팅되지 않고 **첫 hold 요청 시 DB 기준으로 lazy 초기화**된다(좌석 생성 API가 없어 이게 사실상 유일한 초기화 경로 — [architecture.md §6](../architecture.md#6-미확정-항목) 참고).

기존엔 SETNX라 최종 값 자체는 한 번만 세팅됐지만, 값이 없는 짧은 구간에 몰린 동시 요청 **전부가 각자 DB 조회**를 날려 커넥션 풀을 순간 소진시킬 수 있었다(부하테스트에서 5xx로 확인).

---

## 결정

`ensureInventoryInitialized()`에 Redisson 분산락(`lock:inventory-init:{showId}`)을 추가해 초기화용 DB 조회 자체를 1건으로 제한한다. 락을 기다린 나머지 요청은 락 해제 후 이미 세팅된 키를 보고 DB 조회 없이 통과한다.

---

## 이유

- 문제는 "값이 여러 번 쓰이는 것"(SETNX가 이미 방어)이 아니라 "DB 조회가 N번 나가는 것"이었다 — 방어 지점을 조회 앞으로 당겨야 했다.
- 대기열 통과 직후 트래픽이 한 순간에 몰리는 서비스 특성상, 첫 요청 구간의 동시성이 일반 서비스보다 훨씬 높다.

## 재검토 조건

좌석 생성 API가 생기면 생성 시점에 inventory를 미리 세팅하는 방식으로 바꾸고, 이 lazy 초기화 경로(락 포함)는 폴백으로 강등하거나 제거를 검토한다.
