# 260623 구매제한(구매한도 조회) AI 작업 검증 가이드

플랜: `구매한도 조회 API 구현` (1인당 구매 한도 2→4, `GET .../purchase-limit` 추가, `seat/controller` → `seat/presentation/controller` 이동)

## 중점 확인 포인트

1. **`MAX_PER_USER` 값 변경이 실제로 적용됐는지**
   - `SeatService.java`의 상수가 `4`로 바뀌었는지
   - `hold()` 호출 시 그 값이 Lua 스크립트 `ARGV[1]`로 정확히 전달되는지
   - (AI가 상수만 바꾸고 호출부를 누락하는 실수가 흔함)

2. **`getPurchaseLimit` 계산 로직**
   - Redis에 카운트가 없을 때(`null`) `0`으로 처리되는지
   - `remaining`이 음수로 내려가지 않고 `max(limit - purchased, 0)`으로 클램프됐는지

3. **컨트롤러 패키지 이동 후 깨진 import 없는지**
   - `seat/controller` → `seat/presentation/controller`로 옮기면서 다른 파일(테스트, 다른 서비스에서 참조하는 곳 등)에 옛 패키지 경로가 남아있지 않은지 `grep`으로 확인

4. **인증 범위**
   - `@CurrentIdCard`로 가져온 `userId`로만 조회되는지
   - 혹시 AI가 자기도 모르게 path/query parameter로 다른 userId를 받게 만들지 않았는지 (본인 조회만 허용해야 함)

5. **기존 `hold`/`releaseHold` 동작에 회귀(regression)가 없는지**
   - 카운트 증감 로직(`INCR`/`DECR`)을 건드리지 않았는지
   - 기존 테스트(`SeatServiceTest`)가 여전히 통과하는지

6. **TODO 주석이 실제로 코드에 남아있는지**
   - `// TODO: api 엔드포인트 설계 괜찮은지 검토 필요`가 빠지지 않았는지 (검토 대상 표시용)

## 검증 방법

1. `./gradlew :ticketing-service:test` — `SeatServiceTest`의 `getPurchaseLimit` 케이스(구매 0건 시 `limit=4/purchased=0/remaining=4`, 카운트 있을 때 정확히 차감) 포함 전체 통과 확인
2. `./gradlew :ticketing-service:compileJava` — 패키지 이동 후 import 깨짐 없는지 확인
3. 로컬 실행 후 `GET /api/v1/tickets/shows/{showId}/purchase-limit` 호출 → `{limit:4, purchased:0, remaining:4}` 확인
4. `hold` 반복 호출로 구매 수량 증가 후 같은 API 재호출하여 `purchased`/`remaining` 갱신 확인, 4장째 이후 `hold` 호출 시 `PURCHASE_LIMIT_EXCEEDED` 재확인
