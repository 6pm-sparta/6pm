## 병목 의심 지점

1. **`QueueScheduler.findActiveShowIds()` — Redis `KEYS` 명령 사용**
   - O(N) 전체 키 스캔, Redis 싱글 스레드 특성상 다른 명령(좌석 Hold 등)을 블로킹
   - 현재는 동시 활성 공연 수가 적어 체감 영향 적지만, 키 스페이스가 커지면 선착순 공정성에 영향 줄 수 있음
   - TODO.md에 기록되어 있던 알려진 제약 — 우선순위 재검토 필요 (`SCAN` 커서 기반 전환 검토)

2. **`QueueScheduler`에 분산 락 없음**
   - 멀티 인스턴스 환경에서 같은 배치 중복 처리 가능
   - `issue()`(SETNX)/`ZREM`이 멱등이라 데이터 정합성은 깨지지 않으나, 로그/SSE 중복 호출 비효율 발생
   - 인스턴스 수가 늘어나는 시점에 재검토 필요

3. **`SeatService.hold()` 트랜잭션 커밋 지연 윈도우**
   - `assignOrder` 저장과 owner 키 `CONFIRMED` 갱신 사이, `@Transactional` 커밋 전 release가 들어오면 이론상 레이스 재현 가능
   - 윈도우가 매우 작아(로컬 DB 커밋 지연 수준) 우선순위 낮음으로 분류돼 있었음 — 재확인 필요
