# E2E 점검 및 데모 시나리오 (2026-06-24)

> MVP 시연 준비를 위해 `origin/develop` 기준으로 ticketing-service 중심 E2E 흐름을 코드/테스트/문서 3단 대조한 결과.

## 1. 데모 시나리오 (정상 → 예외 → SAGA 보상)

| # | 단계 | API / 트리거 | 기대 결과 |
|---|---|---|---|
| 1 | 대기열 진입 | `POST /api/v1/tickets/shows/{showId}/queue` | 대기열 등록, SSE로 순번 안내 |
| 2 | 토큰 발급 | (백그라운드) `QueueScheduler` | READY 이벤트 + `purchase-token` 발급 |
| 3 | 좌석 목록 조회 | `GET /api/v1/tickets/shows/{showId}/seats` | 토큰 없이도 조회 가능 (AVAILABLE/HOLDING) |
| 4 | 토큰 없이 선점 시도 | `POST .../seats/{seatId}/hold` (토큰 미발급 유저) | `PURCHASE_TOKEN_NOT_FOUND` (403) — 실패 케이스 |
| 5 | 정상 선점 | `POST .../seats/{seatId}/hold` (토큰 보유) | 주문 생성(PENDING), `orderId` 반환 |
| 6 | 구매 한도 초과 | 동일 유저가 한도 초과 선점 시도 | `PURCHASE_LIMIT_EXCEEDED` |
| 7 | 결제 승인 | Kafka `order.payment.completed` | 좌석 BOOKED 전환 + `ticketing.seat.booked` 발행 |
| 8 | 결제 실패/취소 | Kafka `order.payment.failed` / `order.payment.cancelled` | 좌석 해제 + 재고 복구 (SAGA 보상) |

## 2. 설계 문서 vs 실제 코드 대조 결과

### ✅ 일치 확인
- 좌석 목록 조회는 토큰 검증 없음, Hold 시점에만 `purchaseTokenService.exists()` 체크 (`SeatController`, `SeatServiceTest.hold_purchaseTokenNotFound`)
- 대기열 → 토큰 발급 플로우 (`QueueScheduler` 테스트로 검증됨)
- 결제 완료/실패 SAGA 보상 (`SeatConfirmService.confirmSeat/releaseSeat`, Kafka 토픽 양방향 구현 확인)
- hold ↔ release 동시 호출 레이스 방지 (owner 키 PENDING/CONFIRMED 상태값으로 차단)

### ✅ 해소 확인 (2026-07-01)
- **`OrderClient` 호출 경로 불일치** — 해소됨. 양쪽 다 `/internal/v1/orders`로 일치 (`OrderClient.java`, `InternalOrderController.java`).
- **주문 취소 연동** (#155) — 해소됨. `releaseHold()`가 `OrderClient.cancel()` 호출, order-service 타임아웃 자동취소(#231)의 `order.hold.released`를 ticketing `PaymentEventConsumer.onHoldReleased()`가 구독해 좌석 해제. 자세한 내용은 `CHANGELOG.md` 2026-07-01 항목 참고.

### ❌ 남아있는 미구현 항목
- `QueueScheduler` 분산 락 없음 — 멀티 인스턴스 환경에서 중복 처리 가능 (멱등 처리로 정합성은 깨지지 않음, 우선순위 낮음)
- `findActiveShowIds()`가 `KEYS` 명령(O(N) 전체 스캔) 사용 — 키 스페이스 커지면 `SCAN` 전환 필요
