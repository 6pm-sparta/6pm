# 대기열 통과 시 구매 토큰 발급 설계 (2026-06-23)

## 배경
- 브랜치: `feature/57-purchase-token`
- 대기열(`waiting_queue:{showId}`)을 통과한 사용자에게 좌석 구매 자격을 부여하는 "구매 토큰" 발급/검증 설계
- 현재 `QueueScheduler`에 임시 구현이 있으나(키 네이밍, 검증 로직 부재) 정리 필요

## auth-service 토큰 패턴 조사 결과
- Access/Refresh Token: JWT(HS256), Redis 키 `refresh:{userId}:{tokenId}`, `blacklist:access:{tokenId}`
- 검증: Gateway가 JWT 파싱 → `UserIdCard` 생성 → `X-Id-Card` / `X-Id-Card-Signature` 헤더로 하위 서비스 전달 → `IdCardVerificationFilter`에서 재검증

## 구매 토큰 설계 결론
- **랜덤 토큰 문자열/JWT 불필요**: 호출자 신원은 게이트웨이가 `X-Id-Card`로 이미 보장하므로, 구매 토큰은 "자격 존재 여부"만 의미하는 boolean 플래그로 충분
- **Redis 키**: `purchase-token:{showId}:{userId}` (값 `"1"`, TTL 600초)
  - 기존 `purchase-token:{userId}`는 showId가 빠져 있어 한 유저가 여러 공연 대기열에 동시에 있을 때 충돌 가능 → 키에 showId 포함 필수
- **클라이언트 알림**: 기존 순번 브로드캐스트용 SSE(`QueueSseService`) 채널에 `ENTERED` 이벤트 추가 (토큰 값 전달 불필요, 발급 사실만 알리면 됨)
  - SSE 연결 끊김 대비: `/queue/status` 폴백 API에서 토큰 발급 여부 확인 가능하게

## 흐름
1. `QueueService.enter()` — ZSET에 진입 (기존 구현)
2. `QueueScheduler.processShowQueue()` — 배치로 ZSET에서 꺼내 `purchase-token:{showId}:{userId}` SETNX, TTL 600초
3. SSE `ENTERED` 이벤트 전송
4. 좌석 Hold API 진입 시 `EXISTS purchase-token:{showId}:{userId}` 체크 (현재 부재 → 추가 필요)
   - 없음 → 403 / 대기열 재진입 유도
   - 있음 → 정상 진행
5. TTL 만료 시 자동 삭제 (auth의 blacklist처럼 명시적 무효화 로직 불필요)

## 검증 위치
- 현재 검증이 필요한 지점이 좌석 Hold 하나뿐 → 필터/인터셉터 대신 `SeatHoldService` 내부에 직접 체크하는 방식 우선 채택
- API가 늘어나면 AOP/인터셉터로 추출 고려

## 미결 엣지케이스
- 스케줄러 멀티 인스턴스 시 배치 중복 처리 가능성 → 분산 락 필요 여부 검토
- TTL 만료 후 재진입 정책 (재발급 vs 대기열 맨 뒤로)
- 좌석 Hold 성공 후 토큰 즉시 삭제 여부 (결제 재시도 가능성 고려 시 유지 권장)
