# order-service 부하테스트 진행상황

> 작성: 승현 · 최종 업데이트: 2026-07-06 · 상태: 진행 중 (로컬 baseline 확보, 원인 미확정)

## 목적

SLO 정의서 기준 order-service 관련 목표(SLO-4: 결제 성공률 ≥99%, 공통 인프라: DB 커넥션풀 고갈 여부)의
baseline 수치를 확보하고, 부하 상황에서의 병목 지점을 식별한다.

## 테스트 구성

- 시나리오: 주문생성(내부 호출) → 결제요청(외부 진입점, 게이트웨이 경유) → 결제결과 폴링 확인
- 도구: k6, 단계적 VU 증가(20 → 300 → ...)
- 결제 승인/실패/지연을 확률적으로 재현하는 모드를 켜서, 실제 결제 성공률에 가까운 조건으로 측정

## 진행 상황

### 1. 20 VU — 정상 baseline 확보

- 실패율 0%, 지연시간 매우 낮음 (주문생성 p99 ~38ms, 결제요청 p99 ~77ms)
- 결제 성공률 95.46% — 확률적 장애 주입 설계값(약 95%)과 정확히 일치
- 이 규모에서는 부하로 인한 왜곡 없이, 설계한 시나리오 그대로 재현됨을 확인

### 2. 300 VU — 심각한 병목 발견

- 전체 요청 실패율 약 49%
- 결제요청 대부분이 게이트웨이 단계에서 즉시 거부됨 (서비스 일시 불가 응답) — 실제 결제 처리 로직에
  도달하기 전에 막히는 것으로 확인
- 누적 기준 생성된 주문의 97.6%가 결제 시도 자체를 못 해보고 대기 상태로 남음
- 결제 결과 확인 단계에서 "응답 없음"으로 분류되는 비율이 설계값(2%) 대비 약 19배(38.7%)로 치솟음 —
  확률적 설계가 아니라 실제 처리 지연이 원인으로 판단됨

### 3. 원인 조사 — 두 가지 가설을 검증했으나 기각

| 가설 | 조치 | 결과 |
| --- | --- | --- |
| 결제 결과 통보(webhook) 처리 스레드 부족 | 스레드 수 4 → 20으로 증설 후 재측정 | 개선 없음. 지연시간·실패율 동일 수준 유지 |
| DB 커넥션 풀 부족 | 풀 크기 10 → 30으로 증설 후 재측정 | 개선 없음. 대기 중인 커넥션 요청이 여전히 최대 63~114건까지 관측됨 |

두 조치 모두 실제 병목이 아니었거나, 다른 병목에 가려져 효과가 드러나지 않은 것으로 보인다.

### 4. 분산 추적(Zipkin)으로 원인 추적 시도 — 계측 공백 발견

느린 요청의 세부 구간(DB 조회, 캐시 조회 등)을 나눠서 보려 했으나, 현재 하나의 요청이 단일 구간으로만
기록되고 있어 내부 어느 지점이 느린지 구분이 안 됨. **원인 규명 전에 계측을 먼저 보강해야 하는 상황.**

## 현재 결론

- 300 VU 기준 병목의 정확한 원인은 아직 미확정
- 다만 두 차례 조치가 모두 효과 없었다는 것 자체가 단서 — 더 근본적인 지점(쿼리/락 경합, 공유 캐시
  응답 지연 등)일 가능성에 무게를 두고 있음
- **로컬 환경 한 대에서 전체 서비스 + 부하 도구를 함께 실행 중**이라, 관측된 수치 중 일부는 순수 자원
  경쟁(로컬 머신 한계)에서 비롯된 것일 수 있음 — 이번 수치를 절대 기준으로 삼지 않기로 함
