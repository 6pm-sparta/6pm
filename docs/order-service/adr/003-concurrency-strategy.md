# ADR 003 — 비관적 락 + Redis 분산락 이중 방어 전략

> 📌 **상태명 변경 참고**: 이 문서는 (주문 상태 머신 재설계) 이전에 작성되어 `PAYMENT_REQUESTED`/`PAID`/`REFUND_REQUESTED`/`REFUNDED`/`COMPENSATING` 같은 옛 상태명을 그대로 쓴다. 결정 배경을 남긴 역사적 기록이라 원문은 유지하고, 현재 상태명 매핑은 [architecture.md](../architecture.md) 3번 섹션을 참고할 것.

**날짜**: 2026-06  
**상태**: 확정

## 배경

두 가지 동시성 문제가 있다.

**결제 요청 동시성**
결제 요청 API에 동시에 여러 요청이 들어올 경우 중복 결제가 발생할 수 있다. 특히 서버 인스턴스가 여러 개인 환경에서 DB 비관적 락만으로는 PG API 중복 호출을 막지 못한다.

**주문 상태 동시 변경**
유저 직접 취소와 Kafka 이벤트(좌석 확정, 좌석 예매 실패)가 거의 동시에 같은 주문을 건드릴 수 있다. 예를 들어 유저가 PAID 상태에서 취소 요청을 보낸 거의 같은 시점에 `ticketing.seat.booked` 이벤트가 도착하는 경우, 둘 다 처리되면 CONFIRMED이면서 동시에 취소 흐름으로 빠지는 정합성 오류가 생긴다.

---

## 결정: Redis 분산락(1차) + DB 비관적 락(2차) 이중 방어

---

## 이유

**낙관적 락을 거부한 이유**

낙관적 락은 충돌 감지 후 재시도를 전제로 한다. 결제 도메인에서 재시도는 PG API 재호출로 이어질 수 있어 중복 결제 위험이 생긴다.

구체적으로, 결제 요청 흐름에서:

```
1. 주문 조회 (version=1 확인)
2. PG API 호출 (외부 호출, 트랜잭션 밖)
3. 주문 상태 업데이트 시도 → 충돌 감지
4. 재시도 → PG API 재호출 발생
```

3번에서 충돌이 감지되면 재시도 시 PG를 다시 호출하게 되어 중복 결제가 발생할 수 있다. 결제 도메인에서 충돌 후 재시도는 허용할 수 없다.

주문 상태 변경(취소, 좌석 확정 등)도 마찬가지다. 충돌 후 재시도 시 이미 상태가 바뀐 주문에 잘못된 전이가 일어날 수 있다. 비관적 락으로 한쪽만 먼저 통과시키고, 나중에 들어온 쪽은 변경된 상태를 보고 분기(멱등 처리 또는 409)된다.

**Redis 분산락만으로 충분하지 않은 이유**

Redis 장애 시 분산락이 동작하지 않는다. DB 비관적 락이 최종 안전망 역할을 해야 한다.

**DB 비관적 락만으로 충분하지 않은 이유 (결제 요청)**

DB 비관적 락은 동일 DB 트랜잭션 내에서만 보호한다. 트랜잭션 커밋 후 PG API 호출 구간에서는 락이 이미 해제된 상태라 다른 인스턴스의 동시 요청이 PG를 중복 호출할 수 있다.

---

## 락 획득 순서와 역할 분리

**결제 요청**
```
1. Redis 분산락 획득   ← 인스턴스 간 동시 요청 1차 차단
2. 주문 비관적 락 조회 ← DB 레벨 상태 검증 최종 방어
3. 상태 검증 + 전이 + 커밋
4. 분산락 해제         ← 커밋 후, PG 호출 전에 해제
5. PG API 호출
```

4번에서 상태가 이미 PAYMENT_REQUESTED로 커밋된 뒤 락을 해제한다. 락 해제 후 들어오는 동시 요청은 상태 검증(PENDING만 통과)에서 자연히 거부된다.

**주문 상태 변경 (취소, 좌석 확정, SAGA 보상)**
```
1. 주문 비관적 락 조회 ← 분산락 없이 비관적 락만 사용
2. 상태 검증 + 전이 + 커밋
```

취소/확정/보상은 PG 외부 호출이 트랜잭션 밖에서 일어나지 않으므로 Redis 분산락 없이 비관적 락만으로 충분하다.

---

## 트레이드오프

- Redis 분산락 획득 실패 시 즉시 409를 반환한다. 클라이언트가 재시도할 수 있다.
- Redis 장애 시 분산락이 차단되어 결제 요청이 막힌다. 이 경우 DB 비관적 락이 최종 방어선이 된다.
- 비관적 락 대기 시간이 발생할 수 있다. 락 타임아웃 설정으로 무한 대기를 방지한다.

> ⚠️ **구현 확인 결과 (2026-07) — 설계 의도와 실제 코드 불일치**: 위 두 번째 트레이드오프("Redis 장애 시 DB 비관적 락이 최종 방어선")는 실제로 구현되지 않았다. `PaymentRequestService.requestPayment()`의 `lock.tryLock()` 호출은 `InterruptedException`만 잡고 있어서, Redis(Redisson) 연결 자체가 끊기면 그 예외가 그대로 전파되어 요청이 실패한다(DB 비관적 락 단계에 도달하기 전에 끊김). 즉 결제 요청 경로에서는 Redis가 단일 장애점(SPOF)이다. `redis-keys.md`의 "공통 장애 대응 패턴" 표에도 동일하게 기록해뒀다. 고치려면 `tryLock()`을 감싸는 catch에 Redis 연결 예외 케이스를 추가하고, 이 경우에만 분산락을 건너뛰고 DB 비관적 락으로 직행하는 명시적 폴백 경로가 필요하다.
