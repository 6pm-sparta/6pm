# ADR 004 — payments 1:N 구조 (실패 시 새 레코드 INSERT)

**날짜**: 2026-06  
**상태**: 확정

---

## 배경

결제 재시도 이력을 어떻게 관리할지 두 가지 방식을 검토했다.

**방식 A — retry_count 컬럼**
payments 테이블에 `retry_count` 컬럼을 두고, 실패 시 카운트만 올린다. 레코드는 주문당 1개.

**방식 B — 실패 시 새 레코드 INSERT**
결제 시도마다 새 레코드를 INSERT한다. 레코드는 주문당 N개.

---

## 결정: 방식 B 채택

---

## 이유

retry_count 방식은 "몇 번 실패했는가"는 알 수 있지만, 각 시도의 실패 사유와 PG 트랜잭션 ID를 추적할 수 없다.

결제 도메인에서는 아래 정보가 반드시 필요하다.

- 1차 시도에서 어떤 이유로 실패했는가
- 각 시도에서 PG가 발급한 트랜잭션 ID가 무엇인가 (환불 요청 시 필요)
- 최종 승인된 결제 건이 몇 번째 시도였는가

새 레코드 INSERT 방식은 전체 시도 이력을 그대로 보존하므로 CS 처리, 환불 분쟁, 장애 추적 모두에서 근거를 제공할 수 있다.

---

## 트레이드오프

- 주문당 레코드가 N개가 되지만, 결제 재시도 횟수 자체가 제한적이라 무한정 쌓이지 않는다.
- `GET /orders/{orderId}/payments` 응답이 배열이 되는데, 결제 시도가 없는 주문(PENDING 상태)도 200 + 빈 배열로 응답한다. 절대 404를 반환하지 않는다.
