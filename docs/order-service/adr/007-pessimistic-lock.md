# ADR 007 — 비관적 락 vs 낙관적 락

**날짜**: 2026-06  
**상태**: 확정

---

## 배경

주문 상태 동시 변경을 막기 위해 락 전략을 결정해야 했다. 두 가지를 검토했다.

**낙관적 락 (Optimistic Lock)**
`@Version` 컬럼을 두고, 충돌 시 `OptimisticLockException`을 던진다. 재시도 로직을 호출부에서 처리한다.

**비관적 락 (Pessimistic Lock)**
조회 시점에 DB 레벨 배타 락을 걸어 다른 트랜잭션이 접근하지 못하게 막는다.

---

## 결정: 비관적 락 채택, 낙관적 락 거부

---

## 낙관적 락을 거부한 이유

낙관적 락은 충돌 감지 후 재시도를 전제로 한다. 결제 도메인에서 재시도는 PG API 재호출로 이어질 수 있어 중복 결제 위험이 생긴다.

구체적으로, 결제 요청 흐름에서:

```
1. 주문 조회 (version=1 확인)
2. PG API 호출 (외부 호출, 트랜잭션 밖)
3. 주문 상태 업데이트 시도 → OptimisticLockException
4. 재시도 → PG API 재호출 발생
```

3번에서 예외가 나면 재시도 시 PG를 다시 호출하게 되어 중복 결제가 발생할 수 있다. 결제 도메인에서 낙관적 락의 재시도는 허용할 수 없다.

---

## 비관적 락을 채택한 이유

조회 시점에 락을 잡아 다른 트랜잭션의 진입 자체를 막는다. 충돌 시 재시도 없이 대기 후 순차 처리된다. PG API가 호출되기 전에 상태 검증이 완료되므로 중복 호출 여지가 없다.

---

## 트레이드오프

- 락 대기 시간이 발생할 수 있다. 락 타임아웃 설정으로 무한 대기를 방지한다.
- 다중 인스턴스 환경에서 DB 비관적 락만으로는 PG API 중복 호출을 막지 못한다. Redis 분산락이 인스턴스 간 1차 방어를 담당한다. ([ADR 003](./003-concurrency-strategy.md) 참고)
